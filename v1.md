- **Error model.** One class `UnlighthouseError extends Error` with `.code: string` discriminant. Codes are stable (`NOT_SUPPORTED`, `QUOTA_EXCEEDED`, `ACTIVE_SCAN_CONFLICT`). No class hierarchy.

## Goals

1. **Worker-portable core.** `@unlighthouse/core` must be importable into a Cloudflare Worker (or any non-Node runtime) without dragging in puppeteer, chrome-launcher, better-sqlite3, h3-server, listhen, or lighthouse-npm.
2. **One typed API surface, two hosts.** The dashboard API in `packages/ui/ROADMAP.md` is served identically by the local CLI (Node + listhen) and by the SaaS (Cloudflare Worker + Nitro / Hono). Same routes, same types.
3. **Pluggable seams for crawler, runner, and storage** so SaaS variants (`cloud.unlighthouse.dev`-style) reuse the same primitives instead of reimplementing them.
4. **UI imports types only.** `@unlighthouse/ui` typechecks against `@unlighthouse/contracts` + `@unlighthouse/api/types`. Zero transitive pull on runtime deps.

## Runtime baseline: Node ≥ 22

v1 sets `engines.node: ">=22.0.0"` in every published package. Node 22 is the active LTS line (April 2024 → April 2027); current 0.x supports back to Node 18 which exits LTS April 2025. Dropping <22 unlocks:

- **`require(esm)`** (stable on 22.12+). Dual-publish complexity shrinks: presets can `require()` ESM-only deps (hookable, defu, c12, citty, crossws, unstorage) from CJS bin shims without `dynamic import()` dances.
- **Native `fetch`, `Request`, `Response`, `Headers`, `FormData`** (stable since 21). `@unlighthouse/api` and `@unlighthouse/core` use the global; no `node-fetch` / `undici` import in Node-only paths, identical surface as Workers.
- **`globalThis.WebSocket` client + `node:test`'s `MockAgent`**; `@unlighthouse/api/http` can use the WHATWG WS client where it previously needed `ws`.
- **`node:sqlite`** (stable on 22.5+) is the **v1.0 default driver** for `storage/drizzle` (sqlite dialect). `better-sqlite3` is dropped as a default dep entirely; users who want it can pass their own `Database` instance into `drizzleStorage({ driver })`. **Eliminates the #1 0.x install issue** (better-sqlite3 native build failures). (D-021)
- **`AbortSignal.any([...])`** (stable on 20.3 / 22). Cleaner fan-in for `core.run({ signal })` + per-audit timeout signals without manual listener plumbing.
- **`util.styleText`** for CLI output without picocolors / kleur (small win; presets still likely want consola).
- **`structuredClone`, `Array.fromAsync`, `Promise.withResolvers`** — `CrawlSession.done` Promise becomes one line.
- **Top-level `await` in CJS via `require(esm)`** — bin shims simplify.
- **Zod 4** ships `z.toJSONSchema()` natively. `@unlighthouse/contracts/manifest` becomes a built-in, not a separate subpath; no `zod-to-json-schema` dep. (D-021)

Worker target (CF) is already on V8 with all of these. SaaS Node hosts on 22 too. No back-compat plan.

## Ports (four seams + hookable spine)

```
   SeedSource[]  ──►  Crawler  ──►  Auditor (often AuditorRouter)
                       │  ▲              │
                       │  └─core.run()   └── audits each URL
                       │
                       ▼
                    Storage (rows + blobs)
                       │
                       └──── @unlighthouse/api ──► UI
                                  │
               (Hookable bus: scan:*, assert:*, compare:*, quota:*, log)
```

**v1 ports (≥2 real adapters today, or treeshake-driven):**

1. `SeedSource` — sitemap, manual, route-definitions, fuse (all already in `src/discovery/`)
2. `Crawler` — parallel-map, crawlee, cloudflare-crawl. Importable adapters; user passes one in. Treeshakeable by import boundary.
3. `Auditor` — local, psi, crux at minimum; AuditorRouter composes
4. `Storage` — drizzle (rows) + unstorage (blobs); D1+R2 in CF preset

**Deferred (one adapter today, promote to port when second ships):**

- `Policy` — robots only. Inline shape: `Crawler.run({ allows: (url) => bool, crawlDelayMs })`. Promote when glob policy ships or multi-host scans need per-host policies.
- `BrowserPool` — puppeteer-cluster only. Inline shape: pass `{ acquire, release }` directly to `auditors/local` options. Promote when `pools/browserless-remote` or `pools/crawlee` (shared with crawler) is real.
- `Broadcaster` — in-process only. Inline shape: `core.run()` exposes `events: AsyncIterable<HookEvent>`; the host (CLI listhen / Worker DO) decides fan-out. Promote when DO-broadcaster ships in Phase 5.
- `RateLimiter` — unstorage-counter only. Inline shape: hook bus only; `audit:before` payload includes `quotaBucket`, a listener (`rateLimitedPick` helper for the router) consults whatever counter the host provides. Promote when sliding-window or `rate-limiter-flexible` ships.

Plus the `Hookable` bus as the cross-cutting event spine.

**Why four, not eight (rationale).** The skill rule: one adapter = hypothetical seam; two = real one. `Crawler` is the one exception — it has three real adapters today (parallel-map / crawlee / cloudflare-crawl) and the seam exists primarily for **treeshake**: a Worker bundle must not pull in crawlee, and a CLI bundle must not pull in cloudflare-crawl. An import boundary enforces this cleanly; a `config.crawler` enum inside `core.run()` would force a dynamic-import dance that's fragile. The remaining four deferred seams had a single concrete adapter today and can be promoted SemVer-minor as the second adapter actually ships, via subpath addition to `@unlighthouse/contracts/ports`.

### `SeedSource` — pure URL producers

```ts
interface SeedSource {
  seeds(): AsyncIterable<Seed>
}
interface Seed { url: string; from: 'sitemap' | 'manual' | 'route-def' }
```

Adapters: `seeds/{sitemap, manual, route-definitions, fuse}`. All pure HTTP except `route-definitions` (Node fs scan of Nuxt/Next page files).

### `Policy` (deferred — inline shape, not a v1 port)

Until a second adapter ships (e.g. `policies/glob` for path-level filtering, or multi-host scans needing per-host robots), policy is an inline shape on `core.run()`:

```ts
core.run({ allows: (url) => bool, crawlDelayMs: number | undefined })
```

`robotsPolicy({ siteUrl, ua })` returns `{ allows, crawlDelayMs, discoveredSitemaps }` — a value object, not a port. CLI preset wires `robotsPolicy()` into the run options. The Crawler-internal filter rules below still hold; they just consume the inline functions instead of a `Policy` interface.

**Deferred port spec (for reference when promoted):**

```ts
interface Policy {
  allows(url: string): boolean
  crawlDelay(): number | undefined
}
```

Adapters: `policies/{robots, glob, allow-all, fuse}`.

Design decisions (grilled):

- **Async construction, sync evaluation.** `robotsPolicy({ siteUrl })` returns `Promise<Policy>`; the engine calls `.allows()` thousands of times in tight loops, must be sync.
- **UA bound at construction**, not per-call. Drop `ua` argument from the port; pass it to `robotsPolicy({ ua })`. Add back per-call only if a real multi-UA scan use case appears.
- **`fusePolicy` semantics: AND.** Every policy must allow. `crawlDelay()` returns the max (slowest wins). No OR variant.
- **Fail-open by default.** robots.txt 404 / timeout / malformed → allow-all with a warning on the hookable bus. Optional `strict: true` flips to fail-closed for paranoid CI.
- **Single-host.** Per-host robots caching is not a concern today; if multi-origin scans become a thing, the seed source enumerates origins and each origin gets its own fused policy.
- **`discoveredSitemaps` lives on the concrete adapter, not the port.** `robotsPolicy()` returns `Policy & { discoveredSitemaps: string[] }`. User-land glue feeds those into `sitemapSeeds(...)`. The port stays narrow.
- **Filter applied at two points** inside the Crawler adapter: pre-enqueue for seeds, and inside the request handler for URLs discovered during crawl. Crawl-delay read once at engine start.
- **Hookable events on the adapter, not the port.** `policy:robots:fetched`, `policy:robots:error`. The engine optionally emits `policy:rejected` per URL (gated behind a debug flag — volume-heavy).
- **Edge cases.** Strip fragments before evaluating; query strings match against the path; wildcards (`Disallow: /*.pdf$`) supported via a real parser (robots-parser or hand-rolled).

### `Crawler` — drives the seed→audit loop

```ts
interface Crawler {
  run(opts: {
    seeds: SeedSource
    audit: (url: string, ctx: CrawlCtx) => Promise<void>   // core wires this to the Auditor
    allows?: (url: string) => boolean                       // inline Policy shape
    crawlDelayMs?: number
    signal?: AbortSignal
  }): AsyncIterable<CrawlEvent>                             // url-discovered / url-completed / etc.

  pause?(): Promise<void>                                   // optional capability
  resume?(): Promise<void>                                  // optional capability
  state?(): CrawlerState                                    // 'idle' | 'running' | 'paused'
}
```

The user supplies a `Crawler` to `createUnlighthouseCore({ crawler })`. `core.run()` owns `scanId`, the `AbortController`, and the translation from `CrawlEvent` → stable `scan:*` events on the `HookMap`. The crawler itself emits **adapter-private** events via the `debug?` callback.

**Pause/resume is a capability, not a contract.** Adapters that can pause (parallel-map, crawlee) implement the optional methods. Adapters that can't (cloudflare-crawl, future queue-bridge, firecrawl) omit them; `CrawlSession.pause()` throws `NotSupportedError`. UI surfaces the capability via `session.capabilities` for affordance gating.

**Three blessed adapters ship under `@unlighthouse/core/crawlers/*`:**

| Adapter | Use case | Pause/resume | Worker-safe |
|---|---|---|---|
| `crawlers/parallel-map` | Finite seed list, no discovery (PSI/CrUX bulk runs) | ✓ (stops pulling from iterator) | ✓ |
| `crawlers/crawlee` | Sitemap + manual + link-discovery URL graph walk | ✓ (crawlee native) | ✗ (Node-only) |
| `crawlers/cloudflare-crawl` | CF Browser Rendering crawl inside a Worker | ✗ throws `NotSupportedError` | ✓ |

`parallel-map` is the trivial no-discovery degenerate case of crawling — used when seeds are fully enumerated upfront and no link-following is needed.

Users wanting a custom strategy (queue-bridge consuming BullMQ / CF Queues / SQS, firecrawl-backed, etc.) implement the `Crawler` interface and pass it in. No PR to core required.

`crawlers/queue-bridge` writes seeds to an external queue and reads results back — a user-land example, not shipped in core.

**Stable `scan:*` events are emitted by `core.run()` orchestration**, not by the crawler. Adapter-internal events use the `debug?` callback (see §Hook bus). This keeps adapters interchangeable — a crawler doesn't know about `scanId` or `HookMap` payload shapes.

Why "Crawler" wins back the name: sitemap moved to `seeds/`, robots moved to `policies/`. The remaining adapters all literally crawl.

### `Auditor` — produces a Lighthouse-shaped report for one URL

```ts
interface Auditor {
  audit(url: string, page?: Page, opts?: { signal?: AbortSignal }): Promise<LighthouseReport>
  readonly capabilities: AuditorCapabilities
}

interface AuditorCapabilities {
  reliablePerfScores: boolean          // false for remote-CDP (network RTT contaminates LCP/TBT/SI)
  reliableFieldData:  boolean          // true for CrUX; false for lab
  supportsThrottling: boolean          // false for fetch-based (PSI, CrUX, dataforseo)
  categories: Category[]               // which Lighthouse categories the adapter can produce
}
```

`capabilities` is **read at construction** and exposed so router `pick` functions can route per-category (e.g. "send perf to local; everything else to cdp-connect"). The router does not enforce them; it's metadata for callers.

Concrete adapters: `auditors/{local, cdp-connect, psi, crux, dataforseo, mock}`.

- **`local`** — spawns Chrome locally (chrome-launcher + Puppeteer). `capabilities.reliablePerfScores = true`. Optional `pool` option for browser sharing.
- **`cdp-connect`** — connects to **any remote Chrome via CDP WebSocket** and runs Lighthouse against the resulting page. Constructor takes `{ browserWSEndpoint, headers? }`. Single adapter covers **Cloudflare Browser Rendering** (Browser Run), **browserless.io**, **self-hosted puppeteer-cluster**, and any future remote-CDP host. `capabilities.reliablePerfScores = false` — see D-022.
- **`psi`**, **`crux`**, **`dataforseo`** — fetch precomputed reports. No throttling control. `crux.capabilities.reliableFieldData = true` (the others lab/synthetic).
- **`mock`** — fixture-driven for tests.

The verb "audit" papers over the difference at the API consumer's layer (they all return a `LighthouseReport`).

**Cloudflare Browser Rendering** specifically wires in two ways (both produce a `cdp-connect` adapter):
1. **Worker binding** (preset-only): `auditors/cdp-connect({ browserWSEndpoint: env.MYBROWSER })` — when running inside a CF Worker that has a Browser Rendering binding. Used by `@unlighthouse/cloudflare`.
2. **Node + REST/CDP**: `auditors/cdp-connect({ browserWSEndpoint: 'wss://api.cloudflare.com/.../browser-rendering/devtools/browser', headers: { Authorization: 'Bearer <API_TOKEN>' } })` — usable from any host (CLI included) without running inside a Worker.

CF-specific quota (10 rps, 10 min/day free / 10 hr/month paid, 10 concurrent baseline, 10 min idle timeout, 120 max concurrency, no XPath) is **not** modelled in core; the `rateLimitedPick` helper consumes a host-supplied bucket (`{ tier: 'cf-browser-run' }`) at the router layer.

**Naming alternative still open:** `Source` / `sources/*` if `Auditor` feels overloaded with assertion vocabulary. Current recommendation: `Auditor`.

#### `AuditorRouter` — composition adapter, also an Auditor

A router *is* an Auditor; core never knows it isn't a concrete one. This unlocks fan-out, quota-aware scheduling, fallback chains, and merge of lab + field data.

The router takes a **single `pick` function** that returns the auditor for a given URL. The strategy enum is replaced by named helpers that return `pick` functions:

```ts
// @unlighthouse/core/auditors/route
function routeAuditors(opts: {
  auditors: Array<{ name: string; auditor: Auditor }>
  pick: (auditors, ctx: { url: string }) => Auditor | Promise<Auditor>
}): Auditor

// Named helpers — each returns a `pick` function
roundRobinPick()
weightedPick({ local: 0.8, browserless: 0.2 })
rateLimitedPick(limiter)                         // consults a quota source
fallbackPick()                                   // first in order; bubbles to next on error in router body
predicatePick((url) => name)
```

`merge` and `race` aren't picks (they invoke ≥2 auditors); they're separate composers that also return `Auditor`:

```ts
mergeAuditors([labAuditor, cruxAuditor])         // run all, combine reports
raceAuditors([a, b])                              // first-to-finish wins
```

**Rationale for open shape:** the named-strategy enum closed extension at the contract level (every new routing idea needs a PR to `core`). A single `pick` slot leaves the contract open; helpers ship as composable functions; users can write their own `pick` without expanding the union literal. Capability routing, multi-region, A/B, sampling-by-host — all become user-land `pick` functions.

### `BrowserPool` (deferred — inline shape, not a v1 port)

Pass `{ acquire, release }` directly to `auditors/local` options. `pools/simple({ size })` returns this shape. Promote to a port when `pools/browserless-remote` or `pools/crawlee` (browser sharing between crawl and audit) is real.

**Deferred port spec (for reference when promoted):**

```ts
interface BrowserPool {
  acquire(opts?: { signal?: AbortSignal }): Promise<{ page: Page; release: () => Promise<void> }>
  close(): Promise<void>
}
```

Adapters: `pools/{simple, puppeteer-cluster, crawlee, browserless-remote}`.

- `pools/simple` — in-house. N browsers, FIFO task queue, AbortSignal-aware, hookable events. ~200 lines. Replaces the need to fork `puppeteer-cluster`. **Default for local CLI.**
- `pools/puppeteer-cluster` — back-compat wrapper.
- `pools/crawlee` — exposes crawlee's `BrowserPool` so crawl-time pages and audit-time pages share browsers.
- `pools/browserless-remote` — pool of remote browserless sessions for CI fan-out.

**Decision: do not fork puppeteer-cluster.** It's been unmaintained since 2022, has type issues (today's `cluster.d.ts` vendor file exists because of it), no AbortSignal, no observability. Ship `pools/simple` as the in-house replacement and let `pools/puppeteer-cluster` exist only as a compat shim.

### `RateLimiter` (deferred — hook-bus shape, not a v1 port)

Quota is consulted via a hook listener and the `rateLimitedPick` router helper. The `audit:before` hook payload carries `quotaBucket`; the listener (or `pick` function) consults whatever counter the host supplies. CLI default: an unstorage-memory counter. Worker: hits a Durable Object directly inside the `pick`. No port surface for v1.

**Deferred port spec (for reference when promoted):**

```ts
interface RateLimiter {
  check(bucket: string): Promise<{ allowed: boolean; resetAt?: number }>
  consume(bucket: string, n?: number): Promise<void>
  remaining(bucket: string): Promise<{ remaining: number; limit: number; resetAt: number } | null>
}
```

`remaining()` is dashboard-surfaceable (UI shows "PSI quota: 12,431 / 25,000 today").

**Primitive: unstorage-backed counter with TTL.** Single adapter (`rateLimiters/unstorage`) inherits every unstorage backend (memory, KV, R2, Redis, etc.). Fixed-window semantics; sufficient for free-tier quota tracking. If sliding-window or token-bucket becomes important, swap in `rate-limiter-flexible` as a second adapter without changing the port.

### `Broadcaster` (deferred — `core.run()` exposes events directly)

`core.run()` returns a `CrawlSession` with `events: AsyncIterable<HookEvent>` and `subscribe(handler)`. The host decides how to fan out:

- CLI listhen: in-process subscribe + crossws upgrade for the UI.
- CF Worker preset: forwards each event to a Durable Object (`SCAN_EVENTS_DO.fetch('/broadcast', ...)`).
- CI: drops them (no UI consumer).

No `Broadcaster` port. Promote when there's a real polymorphic consumer of broadcasting in core (today the consumer is always the host, which knows its own runtime). Phase 5 may justify it; revisit then.

**Deferred port spec (for reference when promoted):**

```ts
interface Broadcaster {
  emit(event: WsEvent): Promise<void>     // typed WS events from contracts
  subscribe(handler: (event: WsEvent) => void): () => void   // returns unsubscribe
  close?(): Promise<void>
}
```

**Primitive: crossws** (unjs cross-runtime WS) for transport. Adapters: `broadcasters/{in-process, crossws, durable-object, redis-pubsub}`.

Why a port: Workers can't hold long-lived WS the way Node can. crossws abstracts the WS upgrade across Node, Bun, Deno, CF Workers, Vercel; we layer pub/sub semantics on top per adapter (in-process EventEmitter, Durable Object with WebSocket hibernation, Redis pub/sub for multi-instance fan-out).

Consumed by `@unlighthouse/api/ws`.

### Storage repository signatures (pinned)

```ts
interface ScanRepository {
  create(scan: ScanInsert): Promise<Scan>
  get(scanId: string): Promise<Scan | null>
  update(scanId: string, patch: Partial<ScanInsert>): Promise<Scan>
  findPrevious(q: { site; device; branch?; excludeScanId? }): Promise<Scan | null>
  list(q: ListQuery): Promise<Paginated<Scan>>
  delete(scanId: string): Promise<void>
}

interface ScanRouteRepository {
  putBatch(scanId: string, rows: ExtractedMetrics[]): Promise<void>     // hot path
  upsert(scanId: string, row: ExtractedMetrics): Promise<void>           // route.rescan
  listForScan(scanId: string, q?: RouteListQuery): Promise<Paginated<ScanRoute>>
  get(scanId: string, url: string): Promise<ScanRoute | null>
  delete(scanId: string, url?: string): Promise<void>
}

// EventLogRepository removed. Stable HookMap events are persisted as a single
// gzipped JSONL blob per scan at `scans/{scanId}/events.jsonl.gz`. A hookable
// listener `persistStableEvents(storage)` (in core) writes the blob on
// `scan:complete` / `scan:cancelled` / `scan:error`. Query-by-scan is "fetch
// the blob"; query-by-attribute (rare) is "fetch + filter in memory". If a
// real cross-scan query use case appears, add an index table without
// disturbing the blob.

interface BlobStore {
  put(key: string, data: Uint8Array, opts?: { ttl?; contentType? }): Promise<void>
  get(key: string): Promise<Uint8Array | null>
  has(key: string): Promise<boolean>
  delete(key: string): Promise<void>
}
```

Pagination is **page-based** (sqlite/postgres/D1 friendly). Blob key convention: `scans/{scanId}/lhr/{urlHash}.json.gz`. `putBatch` must be transactional.

### `Storage` — rows + blobs split, primitives chosen

Split by data shape. We ship two adapters; each inherits dozens of backends.

| Half | Shape | Primitive | Backends inherited |
|---|---|---|---|
| `scans` + `routes` | Indexed rows, multi-column queries | **Drizzle ORM** | sqlite via **`node:sqlite`** (default, Node ≥ 22, no native compile), libsql/Turso, D1, better-sqlite3 (opt-in, user passes own `Database`), postgres/mysql at v1.x |
| `blobs` | KV (gzip'd LHR, artifacts, cached HTML) | **unstorage** | fs, memory, s3, r2 (binding + HTTP), cloudflare-kv, redis, azure-blob, vercel-blob, netlify-blobs, github, http, null, + more |

Why split: blobs are KV-shaped (unstorage is the obvious primitive, no indexed queries needed). Rows need multi-column indexed queries (`findPreviousScan({ site, device, branch })`), which is a SQL job; drizzle is the universal-driver SQL layer.

Composition examples:

```ts
// Local CLI default (Node 22 — node:sqlite, no native compile)
import { DatabaseSync } from 'node:sqlite'
import { drizzle } from 'drizzle-orm/node-sqlite'
const storage = createStorage({
  rows:  drizzleStorage({ driver: drizzle(new DatabaseSync('~/.unlighthouse/db.sqlite')) }),
  blobs: unstorageBlobs({ driver: fsDriver({ base: '~/.unlighthouse/blobs' }) }),
})

// Cloudflare Worker
const storage = createStorage({
  rows:  drizzleStorage({ driver: drizzle(env.DB) }),                              // D1
  blobs: unstorageBlobs({ driver: cloudflareR2BindingDriver({ binding: env.BUCKET }) }),
})

// Turso + S3 (edge sqlite + multi-cloud blobs)
const storage = createStorage({
  rows:  drizzleStorage({ driver: drizzle(createClient({ url: TURSO_URL, authToken })) }),
  blobs: unstorageBlobs({ driver: s3Driver({ bucket: 'unlighthouse' }) }),
})
```

No `storage/d1-r2` or `storage/postgres-s3` adapters ship — they're literally two lines of user-land composition.

```ts
interface Storage {
  scans: ScanRepository
  routes: ScanRouteRepository
  blobs: BlobStore
}

interface ScanRepository {
  create(scan: ScanInsert): Promise<Scan>
  get(scanId: string): Promise<Scan | null>
  findPrevious(q: {                    // lhci.md: comparison entry point
    site: string
    device: 'mobile' | 'desktop'
    branch?: string
    excludeScanId?: string
  }): Promise<Scan | null>
  list(q: ListQuery): Promise<{ scans: Scan[]; total: number }>
  delete(scanId: string): Promise<void>
}

interface ScanRouteRepository {
  putBatch(scanId: string, rows: ExtractedMetrics[]): Promise<void>
  listForScan(scanId: string): Promise<ScanRoute[]>
  get(scanId: string, url: string): Promise<ScanRoute | null>
}

interface BlobStore {
  put(key: string, gzip: Uint8Array, opts?: { ttl?: number }): Promise<void>
  get(key: string): Promise<Uint8Array | null>
  delete(key: string): Promise<void>
}
```

Adapters:

- `storage/memory` — pure, in `core`
- `storage/filesystem` — Node fs JSON dump, in `core`
- `@unlighthouse/core/storage/drizzle` — sqlite (drizzle + better-sqlite3); CLI default
- (out-of-tree initially) `storage/d1-r2` — Cloudflare; rows in D1, blobs in R2
- (out-of-tree initially) `storage/postgres-s3` — multi-cloud SaaS

The row/blob split matches reality: extracted metrics are small and queryable; gzipped LHRs are large and only fetched on demand. Different backends specialize differently.

## `createUnlighthouseCore` — pinned factory signature

Locked in the doc so depth is verifiable before Phase 1.7 starts. If during implementation a key gets added that's pure private-internal wiring (factory option 1:1 with private field), revisit — it's a shallowing signal.

```ts
// @unlighthouse/core
export function createUnlighthouseCore(opts: UnlighthouseCoreOptions): UnlighthouseCore

interface UnlighthouseCoreOptions {
  config:   UnlighthouseConfig          // already-resolved (host does c12/env merge + imperative rules); validated by Zod inside
  auditor:  Auditor                     // single, may be an AuditorRouter
  seeds:    SeedSource                  // single, may be fuseSeeds([sitemap, manual, ...])
  crawler:  Crawler                     // single; parallel-map / crawlee / cloudflare-crawl / user-supplied
  storage:  Storage
  hooks?:   Partial<UnlighthouseHooks>  // additive subscribers; merged into the bus
  logger?:  ConsolaInstance             // tagged loggers per adapter derived via logger.withTag(name)
}

interface UnlighthouseCore {
  run(opts?: { signal?: AbortSignal }): CrawlSession    // single-session: throws if a session is already in flight
  session(): CrawlSession | null                        // active session or null
  hooks: Hookable<UnlighthouseHooks>
}

interface CrawlSession {
  scanId: ScanId
  events: AsyncIterable<HookEvent>        // typed HookMap union; host fan-outs from here
  subscribe(handler: (event: HookEvent) => void): () => void
  capabilities: { pausable: boolean }     // derived from whether crawler implements pause/resume
  pause(): Promise<void>                  // delegates to crawler.pause(); throws UnlighthouseError({ code: 'NOT_SUPPORTED' }) if absent
  resume(): Promise<void>                 // delegates to crawler.resume()
  cancel(reason?: string): Promise<void>  // wired to AbortController; propagates to Auditor.audit({ signal })
  state(): ScanStatus
  stats(): { discovered; scanned; failed; total }
  done: Promise<{ scanId: ScanId; summary: ScanSummary }>
}
```

**What the factory hides (verified leverage):**
1. `scanId` generation and threading through every event payload, storage row, and blob key.
2. `AbortController` creation + binding the user's optional signal + propagating cancellation to `Auditor.audit({ signal })`.
3. Driving the user-supplied `Crawler` (parallel-map / crawlee / cloudflare-crawl / custom) and wiring `audit` to the resolved `Auditor`.
4. Bridging `Crawler` `CrawlEvent` iteration → typed `scan:*` events on the `HookMap`.
5. Delegating `pause()` / `resume()` to the crawler's optional methods; throwing `UnlighthouseError({ code: 'NOT_SUPPORTED' })` when absent.
6. Threading `logger.withTag(adapterName)` into every adapter (crawler, auditor, seed source, storage) so observability is one wire-up.
7. Calling `storage.scans.create()`, `storage.routes.putBatch()`, and the `persistStableEvents` listener as the orchestration plays out.
8. Enforcing single-session: a second `run()` call throws `UnlighthouseError({ code: 'ACTIVE_SCAN_CONFLICT' })`. Multi-tenant hosts (CF Worker per request) spawn one `Core` per scan; the factory is cheap (no module-level state, no singletons).

**Six keys on the opts surface, ~600 LOC of orchestration behind it.** Depth verifiable.

## Cross-cutting concerns (covered)

| Concern | Where it lives |
|---|---|
| Cancellation | `AbortSignal` threaded through `Crawler.run({ signal })`, `Auditor.audit({ signal })`, `BrowserPool.acquire({ signal })`. Adapters respect it. |
| Pause/resume | Optional `Crawler.pause()` / `resume()` / `state()` capability. `CrawlSession` delegates and exposes `capabilities.pausable`. Adapters that can't (cloudflare-crawl, firecrawl, queue-bridge) omit the methods; session throws `UnlighthouseError({ code: 'NOT_SUPPORTED' })`. |
| Scan identity | `scanId` generated by `createUnlighthouseCore().run()` per invocation; one in-flight session per Core; carried on the session, every event payload, every storage row, every blob key. |
| Per-run artifact isolation | Blob key convention `scans/{scanId}/lhr/{urlHash}.json.gz` — backend decides physical path; CLI writes to `~/.unlighthouse/<scanId>/`, cloud writes to R2 `<scanId>/`. |
| Logging + adapter observability | Each adapter accepts `logger?: ConsolaInstance` in options; factory threads `logger.withTag(adapterName)`. Presets bind reporters via `consola.addReporter()`. **No `debug?` callback, no second hook-bus tier.** (D-018) |
| Adapter-private user hooks | `puppeteer:before-goto`, `authenticate`, etc. live on the adapter's own `Hookable<TAdapterHooks>` instance, not on the core HookMap. Preset wires user-supplied config into the adapter's `adapterHooks` option. (D-019) |
| Caching | Subsumed into Storage. `storage.blobs.get(cacheKey)` before audit; `put(cacheKey, result, { ttl })` after. |
| Config: merge | `defu` only. User file → env → CLI flags → call args, in that order. (D-011) |
| Config: rules | Imperative conditional rewrites (`localhost → disable throttle`, chrome path autodetect, puppeteer cluster sizing) live in `@unlighthouse/unlighthouse/config/resolve.ts` — **host-specific**. Core never runs rules; it receives an already-resolved `UnlighthouseConfig`. (D-020) |
| Config: validation | Zod schemas in `contracts/config`. `createUnlighthouseCore()` throws `UnlighthouseError({ code: 'CONFIG_INVALID' })` on validation failure. |
| Errors | Single class `UnlighthouseError extends Error` with `.code: string` discriminant. Stable codes documented in `contracts/errors`. No class hierarchy. (D-019c) |
| Auth | NOT a port. Wrapped at host layer (CF preset wraps `core/api` handlers with auth middleware). |
| Quota / rate-limit | `audit:before` hook reads `quotaBucket` from payload and consults a host-supplied counter. `rateLimitedPick` helper for the router. No port surface for v1. |
| LH version skew | `LighthouseReport.lighthouseVersion` is required from every Auditor. `core/report/lighthouse.ts` is the single isolation point. |
| Route-name resolution | `seeds/route-definitions` returns `{ seeds, matcher: (url) => routeName | null }`. Core invokes `matcher` at ingest; `routeName` is a column on `ExtractedMetrics` / `ScanRoute`. Replaces today's `Provider.mockRouter`. |

## AI-agent integration

Goal: Unlighthouse is a first-class tool for AI agents (Claude Code, Cursor, Cline, custom agents), not a thing they shell out to and parse text from.

CLI and MCP must have **full parity**. Users technically could drive everything via CLI from an agent, but they want MCP because that's the native shape. So the *command registry* (see [Command registry](#command-registry-the-source-of-truth) below) is the source of truth, projected to three surfaces: HTTP, MCP, CLI.

### Principles

1. **Transport-agnostic handlers.** Defined in `@unlighthouse/api` as `(input, ctx) => Promise<output>`. Mounted on three transports via auto-projection from the command registry.
2. **Structured output is the default in non-TTY contexts.** When `!process.stdout.isTTY` or `--agent` is set, the CLI emits NDJSON with a `$schema` URL and stable codes; no ANSI, no progress bars, no prompts.
3. **Self-describing.** `unlighthouse manifest --json` emits the full surface: subcommands, flags, exit codes, output schema URLs, examples.
4. **Streamable progress.** Hookable events serialized as NDJSON to stdout or a file the agent can tail. Same payloads as WS events.
5. **Structured errors.** `contracts/errors`: stable codes + `{ code, message, cause, suggestion, docsUrl }`. CLI surfaces JSON on `--agent`.
6. **Idempotent subcommands.** Every command is one-shot, no interactivity, deterministic exit codes.
7. **Determinism flags.** `--seed`, `--fixed-time`, sample-size, etc., to make agent CI loops reproducible.
8. **Queryable history.** `unlighthouse query` exposes Storage as a read API for agents asking "what regressed?" without re-scanning.

### New CLI subcommands (agent-shaped)

| Command | Purpose |
|---|---|
| `unlighthouse audit --site … --output report.json --assert …` | One-shot scan + assert + output; non-zero exit on assertion failure. |
| `unlighthouse query [--scan id] [--filter …] [--sql …]` | Read against Storage. NDJSON output. |
| `unlighthouse compare <baseScanId> <currentScanId> [--output md\|json]` | Run comparison; emit Markdown or JSON. |
| `unlighthouse manifest [--json]` | Self-describing surface. |
| `unlighthouse events --scan <id>` | Tail NDJSON event stream for a running scan. |
| `unlighthouse mcp` | Start MCP server on stdio. |

### `@unlighthouse/mcp` — MCP server preset

```
@unlighthouse/mcp                                (peer: contracts, core, api)  ← preset
└── .                              # createMcpServer(): MCP transport over stdio
                                   # bin: unlighthouse-mcp
```

Composes `@unlighthouse/api` (route handlers) + MCP SDK. Auto-generates the MCP tool registry from `contracts/commands`; same handlers as HTTP. Agents add it to their MCP config and call tools directly.

A `bin/unlighthouse-mcp` shell script lives alongside `bin/unlighthouse`. `unlighthouse mcp` CLI subcommand exec()s the same binary for ergonomics.

### Adapter/reporter additions in `@unlighthouse/cli`

- `reporters/ndjson` — JSONL of route results.
- `reporters/agent-summary` — terse Markdown summary auto-emitted when `--agent`.
- `broadcasters/ndjson-stdout` — wraps the hookable bus, prints events as NDJSON.

### Output schemas

All JSON outputs reference a published JSON Schema:

```json

  "$schema": "https://unlighthouse.dev/schema/v1/scan-result.json",
  "scanId": "...",
  "...": "..."
}
```

Schemas live in `@unlighthouse/contracts/schemas/*.json`, published with the package and hosted on the docs site.

### Why this strengthens the architecture (not bolts on)

The transport-agnostic shape of `@unlighthouse/api` was already implied (HTTP for CLI, Fetch for Worker). MCP is just a third transport over the same handlers. Adding it now costs almost nothing if the handlers are pure `(input) => output` functions; adding it later means refactoring the API package.

This also means `@unlighthouse/api` is more valuable than I initially scoped — it's the single source of truth for *every* programmatic surface (HTTP, Worker, MCP, CLI's own programmatic mode).

## Command registry (the source of truth)

Every operation is defined once in `@unlighthouse/contracts/commands` with Zod input/output schemas, then projected to HTTP, MCP, and CLI by ~30-line generators. CI enforces parity.

### Command type

```ts
interface Command<I, O> {
  name: string                          // 'scan.start' — dot-namespaced
  description: string                   // used by --help AND MCP tool description
  input:  z.ZodType<I>
  output: z.ZodType<O>
  streaming?: boolean                   // handler returns AsyncIterable<O>
  examples?: Array<{ description?; input; output }>
  exitCodes?: Record<string, number>    // CLI: structured-error code → exit code
  http?: { method?: 'GET' | 'POST'; path? }   // override default convention
  mcp?:  { name?; hidden? }
  cli?:  { hidden? }
  permissions?: string[]                // for SaaS auth middleware
}
```

Handlers in `@unlighthouse/api` are typed against the registry with `{ [K in CommandName]: Handler<K> }` so missing/extra handlers fail at compile time.

### Projection pattern

```
contracts/commands  ──┬─►  api/http       (h3 routes; consumed by listhen + CF fetch)
                      ├─►  mcp            (MCP tools over stdio)
                      ├─►  cli            (citty subcommands)
                      └─►  api/client     (typed proxy for the UI)
```

Each projection is a Object.entries loop over `commands` + transport-specific dispatch. ~30 lines each.

### Streaming

`streaming: true` commands return `AsyncIterable<O>`. Projection differs:
- HTTP: SSE response (`text/event-stream`)
- MCP: SDK's streaming tool transport (chunked content)
- CLI: NDJSON to stdout
- Client: async iterator over fetch+SSE

### Parity test (CI-enforced)

Two tests. The name-presence check is cheap and catches the easy class of bug; the schema-equality check catches the dangerous one (projector silently dropping `.optional()`, an enum, or a refinement).

```ts
test('every command is exposed via HTTP, MCP, and CLI', () => {
  for (const [name, cmd] of Object.entries(commands)) {
    expect(httpRoutes.has(name)).toBe(true)
    if (!cmd.mcp?.hidden) expect(mcpTools.has(toMcpName(name))).toBe(true)
    if (!cmd.cli?.hidden) expect(cliCommands.has(name)).toBe(true)
  }
})

test('projection schemas match the contract', () => {
  for (const cmd of Object.values(commands)) {
    const contractJSON = zodToJsonSchema(cmd.input)
    expect(httpRoutes[cmd.name].inputSchema).toEqual(contractJSON)
    if (!cmd.mcp?.hidden)
      expect(mcpTools[cmd.name].inputSchema).toEqual(contractJSON)
    // citty doesn't speak JSON Schema; round-trip via the projector's flag map.
    if (!cmd.cli?.hidden)
      expect(cliCommands[cmd.name].flags).toEqual(cittyFlagsFor(cmd.input))
  }
})
```

## Hookable event catalogue

**Single tier.** The `HookMap` is the contract surface. There is no adapter-private second tier.

### Stable contract (typed in `contracts/hooks/HookMap`, schema-versioned)

Projected to WS, MCP `events.subscribe`, CLI `--agent` NDJSON. Adding an event is SemVer-minor; payload changes are SemVer-major.

| Namespace | Events |
|---|---|
| `scan:*` | `created`, `started`, `discovering`, `scanning`, `progress`, `route-complete`, `route-failed`, `paused`, `resumed`, `cancelled`, `complete`, `error` |
| `assert:*` | `passed`, `failed` |
| `compare:*` | `complete` |
| `quota:*` | `exceeded`, `depleted` |
| `audit:*` | `before`, `after` (used by quota / retry middleware) |
| (root) | `log` (level, message, meta) |

### Adapter-internal observability: `debug?` callback

Instead of a second tier on the hook bus, adapters that want to emit diagnostic events (seed source enumeration, crawler URL-discovered, pool acquire/release, storage writes, router pick selection) accept a single optional callback in their factory options:

```ts
function createUnlighthouseCore(opts: {
  // …
  debug?: (event: string, payload: unknown) => void
}): UnlighthouseCore
```

The factory threads `debug` into every adapter it instantiates. Adapters call `debug?.('crawler:url-discovered', { url })` directly — no bus, no schema, no contract commitment. Observability adapters (consola log appender, OpenTelemetry exporter) bind once at the factory site.

**Rationale.** A second hook-bus tier with one in-tree listener (none today, none planned) is a hypothetical seam by the skill rule. A plain callback is observably equivalent, has zero versioning surface, doesn't risk bus saturation, and treeshakes when unused.

### Signatures

All hooks: `(payload: TypedPayload) => void | Promise<void>`. Awaited in sequence by default; `callHookParallel` available for fanout. No return-value mutation — keeps producers decoupled from consumers.

## Decisions log

Captures the load-bearing calls made during design grilling. Each entry: decision, reasoning, what would force a revisit.

### D-001 — Rewrite + flatten `@unlighthouse/db` into `@unlighthouse/core/storage/drizzle/*`

**Decision:** Do not ship `@unlighthouse/db` as a sibling package. **Rewrite** the drizzle layer (dialect-neutralise the schema, factor out the module-level singleton) AND flatten the result into `@unlighthouse/core` as subpath exports.

**Status as of v1 branch (2026-05-13):** `packages/db` (293 LOC, 2 files) still exists and is receiving edits. Today's shape:
- `getHistoryDb(dataDir)` with a module-level `let db = null` cache (forbidden pattern: singleton at module top level — confirmed via `rg "let db" packages/db/src/index.ts`).
- Schema hardcoded to sqlite via `drizzle/better-sqlite3` (no dialect split, native compile required).
- Single subpath `.` export.

This is **not** a move — it's a rewrite. Phase 1.3 owns it. Effort revised upward (5–6 days, not 3–4).

**The rewrite:**
1. Replace `getHistoryDb()` with `drizzleStorage({ driver })` factory. No module-level state. Each call returns a fresh repository set.
2. Schema lives at `schema/sqlite.ts` (`sqliteTable`); postgres twin is **deferred to v1.x** (no real consumer at v1.0; skill rule: 1 adapter = hypothetical seam). Schema layout makes the future split mechanical.
3. Default driver swapped to `drizzle-orm/node-sqlite` (Node 22's stdlib `node:sqlite`). `better-sqlite3` removed from `dependencies`; users who want it pass their own `Database` instance into `drizzleStorage({ driver })`. (D-021)
4. Author drizzle-kit migrations under `migrations/sqlite/`.
5. Move into `@unlighthouse/core/storage/drizzle/*` subpaths. `@unlighthouse/db` is deleted; its consumers (CLI history tracking) import from the new subpaths.

**Why flatten:**
- `drizzle-orm` is Worker-safe and small (~50KB). D1 access requires it; adding it as a `core` peer doesn't break runtime portability.
- Heavy native drivers (`better-sqlite3`, `pg`, `@libsql/client`) are user-imports either way — flattening does not increase any user's installed footprint by one byte.
- `core` already takes `unstorage` as a peer for blob storage; taking `drizzle-orm` for row storage is the same pattern.
- Removing a publish boundary eliminates the version-coordination tax.
- Schemas + adapter + migrations always change together; co-locating them is structurally honest.

**Treeshake invariant intact:** A Worker bundle importing only `core/storage/memory` + `core/storage/unstorage-blobs` does not pull drizzle in (subpath imports + `sideEffects: false`).

**Until Phase 1.3 completes:** do not add new public surface to `@unlighthouse/db`. Any new schema work goes directly into the new dialect-split shape.

**Revisit if:** drizzle-orm grows a hard dependency on Node built-ins, OR a real user need emerges for "Storage without drizzle as a peer dep at all".

### D-002 — `Auditor` over `Source` (final)

**Decision:** Use `Auditor` and `auditors/*` for the thing that produces a Lighthouse-shaped report per URL.

**Reasoning:**
- The verb in `audit(url)` describes what the port produces, not what the adapter literally does. Every adapter returns `LighthouseReport`; the consumer doesn't care how it got it.
- Action-oriented; pairs naturally with `Crawler` (two distinct verbs).
- `Source` is a noun without a verb; reads awkwardly at call sites.
- Slight imperfection (CrUX is field data, not a lab audit) is acceptable because the API consumer receives an audit-shaped result regardless of source.

**Revisit if:** `Auditor` collides with assertion/audit-of-asserts vocabulary in practice and confuses readers.

### D-003 — Storage primitives: drizzle + unstorage

**Decision:** Row storage uses drizzle (any drizzle driver). Blob storage uses unstorage (any unstorage driver). No custom drivers maintained in-tree.

**Reasoning:**
- Drizzle covers sqlite, libsql/Turso, D1, postgres, mysql with one query API.
- Unstorage covers fs, memory, s3, r2, cloudflare-kv, redis, vercel, netlify, github, http with one KV API.
- Together this inherits ~40+ backends with zero adapter code on our side.
- Both projects are Worker-safe and unjs-family (Nitro-aligned).

**Revisit if:** A backend appears that neither covers AND is important enough to write a custom driver.

### D-004 — Routing is composition, not a new port

**Decision:** `AuditorRouter` implements the `Auditor` port. Fan-out, rate limiting, fallback, merge, race, sampling, and predicate routing are all strategies of `routeAuditors()`. No `Router` port.

**Reasoning:**
- The Crawler doesn't care that an Auditor is "really" several auditors with a strategy. Composition over the port keeps the engine layer ignorant of routing.
- `samplingRouter` (the lhci.md `computeMedianRun` case) is just one more strategy, so sampling falls out of the same primitive.
- Adding a port for routing would force every adapter (memory, local, browserless) to know about routing semantics. They don't need to.

**Revisit if:** A routing strategy emerges that needs Crawler-level coordination (e.g. routing decisions based on cross-URL context the Auditor can't see). None known today.

### D-005 — Command registry in `contracts` is the source of truth

**Decision:** Define every operation (scan.start, query, compare, etc.) once in `@unlighthouse/contracts/commands` with Zod input/output schemas. Project it to three transports: HTTP (`@unlighthouse/api/http`), MCP (`@unlighthouse/mcp`), CLI subcommands (`@unlighthouse/cli`). CI enforces parity.

**Reasoning:**
- Three surfaces must stay in lock-step (HTTP for CLI+Worker, MCP for agents, CLI for humans). Hand-maintaining three lists rots within months.
- Each projection is a ~30-line generator over the registry.
- Adding a new transport (gRPC, webhook) is free if the registry is the source of truth.

**Revisit if:** A transport needs request/response shapes the registry can't express (e.g. binary streams not modeled by Zod).

### D-006 — Crawler emits adapter events; core orchestration emits stable events

**Decision:** The `Crawler` port emits adapter-private `crawler:*` events only. The stable `scan:*` event surface (typed `HookMap`) is emitted by `core.run()` orchestration, which wraps a `CrawlSession`, owns the `scanId`, and translates iteration into stable events.

**Reasoning:** Crawler adapters shouldn't know about `scanId`, `HookMap` payload shapes, or storage. Isolating that to core's orchestration layer keeps adapters interchangeable and the stable event surface single-sourced.

### D-007 — Persist stable-tier events only

**Decision:** `Storage` gains an `EventLogRepository` storing only the typed `HookMap` events (the stable contract tier). Adapter-private events stay on the in-process Hookable bus.

**Reasoning:** Stable events are bounded (~10–20 per route × ~500 routes = ~10k/scan, ~1MB). Adapter-private events are volume-heavy and adapter-shape-dependent; persisting them would bloat storage and create a stale schema commitment.

**Revisit if:** A real observability use case needs durable adapter-private events (e.g. forensic debugging of long-running cloud scans).

### D-008 — Drop `route.list`; `scan.results` is canonical

**Decision:** Single command for listing routes of a scan. Agents call `scan.results({ scanId, pageSize: 9999 })` for "give me all".

### D-009 — CF adapters live in `@unlighthouse/cloudflare`

**Decision:** Platform-specific adapters live in platform-preset packages, not `core` subpaths.

**Reasoning:** `core` exposes adapters built on standard primitives (fetch, drizzle, unstorage, hookable, crossws). CF-specific adapters need `@cloudflare/workers-types`, DO classes, and hibernation patterns that are best co-located with the CF preset. Future Vercel/AWS presets follow the same pattern.

### D-010 — c12 lives in `@unlighthouse/cli/config`

**Decision:** Configuration loaders are host-specific. CLI uses c12 + env + flag overlays. Cloudflare uses env binding + request body. Both produce a validated `UnlighthouseConfig`. Core never imports c12.

### D-011 — defu owns merging; Zod is validation-only; defaults are a literal const

**Decision (inverted from prior draft).** Defaults live in a single `defaultConfig` const in `@unlighthouse/contracts`. defu does all merging — both layered config (user file → env → CLI flags → call args) and per-scan input over `config.scanner`. Zod schemas have **no** `.default()` and run after merge as validation only. `manifest` command emits `defaultConfig` directly (no schema introspection needed).

**Reasoning:**
- Today's code (`packages/unlighthouse/src/unlighthouse.ts:81`) already does `defu(userConfig, config)` with no Zod defaults. Adding Zod `.default()` would *introduce* a dual-merge mechanism, not consolidate one.
- Two merge strategies = exactly where deep-merge / array-replace / null-clears semantics diverge silently. One mechanism, documented semantics (whatever defu does).
- `scanner.exclude = []` semantics now well-defined: defu keeps user's `[]` (array replace by user-provided), defaults supply `[]` if absent.

**Revisit if:** a config field needs Zod-only default semantics that defu can't express — none known today.

### D-012 — `unlighthouse` (alias) owns the bins; `@unlighthouse/cli` does not

**Decision:** The `unlighthouse` npm name stays as a thin re-export of `@unlighthouse/cli` AND **exclusively** owns `bin/unlighthouse` + `bin/unlighthouse-ci`. `@unlighthouse/cli` has **no** `bin` field — it exposes only the programmatic API (`runCli(opts)`, `runCi(opts)`, `createCliPreset()`).

**Reasoning:** Two packages declaring the same `bin` name collide on the user's PATH after `npm i`. Putting bins on the alias keeps the install story for 0.x users intact (`npm i unlighthouse` still gives them `unlighthouse` on PATH) without forcing `@unlighthouse/cli` to also be a CLI executable. Bin shells become 5-line `import('@unlighthouse/cli').then(({ runCli }) => runCli())` — the CLI factory is testable as a function, the bin is a shim.

**Revisit if:** users start `npm i @unlighthouse/cli` directly and expect the binary — at which point keep the bin only on the alias and document the install path.

### D-014 — `Crawler` is a v1 port; treeshake is the seam-justifier

**Decision (supersedes 2nd-pass folding):** Re-promote `Crawler` to a v1 port. Ship three blessed adapters as importable subpaths — `@unlighthouse/core/crawlers/{parallel-map,crawlee,cloudflare-crawl}` — and require the user to pass one via `createUnlighthouseCore({ crawler })`.

**Reasoning:**
- The 2nd-pass folding put strategy selection in `config.crawler` enum branches inside `core.run()`. That forces a `dynamic import('crawlee')` dance to keep Worker bundles slim, which is fragile and obscures the dependency.
- Three real adapters exist today (parallel-map / crawlee / cloudflare-crawl). The seam is real, even if pause/resume capability isn't universal.
- An import boundary enforces treeshake declaratively: a Worker bundle that imports `crawlers/cloudflare-crawl` literally cannot reach crawlee.
- Matches harlan's stated v1 goal of "pluggable crawler implementation." Users supplying a custom crawler (queue-bridge, firecrawl-backed) no longer need a PR to core.
- Pause/resume becomes a **capability** on the port (optional methods) rather than a uniform contract. `CrawlSession.capabilities.pausable` reflects the active adapter. Adapters that can't pause throw `NotSupportedError`. This is the same pattern as `Hookable` opt-in semantics — honest about what each adapter supports.

**Revisit if:** the three blessed adapters consolidate into one (unlikely — Worker vs Node split is permanent).

### D-013 — Runtime-mode discriminator replaced by preset injection

**Decision:** Today's `provider?.name === 'ci' | 'cli'` string branches inside core (e.g. `ws = provider?.name === 'ci' ? null : new WS()` in `unlighthouse.ts:124`) are replaced by **preset injection**. Each preset (`@unlighthouse/cli`, `@unlighthouse/cloudflare`, `@unlighthouse/server`, `@unlighthouse/mcp`) wires the right adapters and event sinks for its runtime; `core` never asks "what mode are we in."

**Reasoning:**
- Branching on a runtime-mode string inside core is a confessed coupling: core knows about every preset that ever existed.
- With three v1 ports (`SeedSource`, `Auditor`, `Storage`) and inline shapes for the rest, the preset just supplies the right factory args. The CI preset passes no event sink; the CLI preset passes an in-process subscriber + crossws WS; the Worker preset passes a DO-forwarder. Core consumes one shape.
- Removes ~6 string-discriminator checks scattered across today's `src/`. Each one becomes "the preset didn't wire that adapter, so the feature is absent." Honest.

**Phase 1 implication:** the `provider` arg to today's `createUnlighthouse(userConfig, provider)` is removed. Presets pass everything explicitly. The 55-site `useUnlighthouse()` unwiring (Phase 1.7) handles this simultaneously — every reach-back for `ctx.provider?.name` becomes a behaviour the caller passed in.

**Revisit if:** a runtime *detection* (not selection) is actually needed inside core for something the preset can't decide — none known today.

### D-015 — Collapse `@unlighthouse/cli` into the `unlighthouse` npm name

**Decision:** No standalone `@unlighthouse/cli` package. The `unlighthouse` package owns bins, programmatic exports (`runCli`, `runCi`, `createCliPreset`), c12 config loading, listhen wiring, reporters, and imperative config rules.

**Reasoning:**
- Per the v1 dep graph, **no other package peer-deps `@unlighthouse/cli`**. Presets (`mcp`, `cloudflare`) peer-dep `contracts + core`. The cli package's only consumer is the alias re-exporting it. That's zero independent consumers; fails the workspace-package threshold from the skill (≥2 independent consumers + zero coupling back to host).
- D-012's bin-conflict problem (two packages claiming the `unlighthouse` bin name) is solved by **not having two packages**. Bin shells become 5-line `import('./dist/cli/runCli.mjs').then(...)` inside `unlighthouse`.
- One fewer publish boundary; one fewer peer-dep chain; one fewer SemVer commitment.

**Revisit if:** a preset materializes that needs `runCli` independently of `unlighthouse`'s bin lifecycle (unlikely — CF Worker doesn't run CLIs).

**Supersedes D-012.**

### D-016 — Collapse `@unlighthouse/api` into `@unlighthouse/core/api/*`

**Decision:** No standalone `@unlighthouse/api` package. Transport-agnostic handlers, HTTP projection, typed UI client, and ApiRoutes types live under `core/api/{handlers,http,types,client}` subpaths.

**Reasoning:**
- `api` peer-depped `core`. Per the skill threshold (≥2 independent consumers **AND zero coupling back to the host**), `api` failed the coupling clause.
- Same set of consumers as `core` (every preset that wants HTTP needs both).
- UI imports types only via `core/api/types` — works identically as a subpath.
- Subpath `+ sideEffects: false` preserves the treeshake invariant: Worker bundles importing `core/api/http` don't pull `listhen`.

**Revisit if:** `api` grows a runtime dep `core` can't take.

### D-017 — `UnlighthouseCore` is single-session

**Decision:** A `Core` instance holds **at most one in-flight `CrawlSession`**. `run()` throws `UnlighthouseError({ code: 'ACTIVE_SCAN_CONFLICT' })` if called while a session is active. Multi-tenant SaaS spawns one `Core` per request (factory is cheap; no module-level state).

**Reasoning:**
- Today's reality (one scan, one process) maps cleanly to single-session. Multi-session would force every hook payload to carry `scanId` and complicate subscribe semantics; the doc had this as latent ambiguity (Core exposed scanId-keyed `state()`/`cancel()` but hooks were not scanId-discriminated).
- Cloud preset wraps one Core per request inside the Worker's `fetch` handler. The Core's lifecycle = the request's lifecycle. DO holds session state across the WS lifetime; Core is GC-able after orchestration completes.
- Dropping scanId-keying simplifies `subscribe`, `state`, `cancel`. The active session is the only session.

**Revisit if:** a single host needs to drive multiple parallel scans within one process AND share storage/hook state across them (CI fan-out across sites — but that's better modelled as N Cores).

### D-018 — `logger?: ConsolaInstance` replaces `debug?` callback

**Decision:** Drop the `debug?: (event, payload) => void` callback proposed in earlier passes. Each adapter accepts `logger?: ConsolaInstance` in its factory options. `createUnlighthouseCore({ logger })` threads `logger.withTag(adapterName)` into every adapter it instantiates. Observability composes via `consola.addReporter()` at the preset.

**Reasoning:**
- The `debug` callback proposed a new untyped event primitive (`(event, payload: unknown) => void`) parallel to the typed HookMap. Two observability primitives for one concern.
- Consola is already in unlighthouse's dep graph for CLI logging. `.withTag()` provides exactly the scoping a per-adapter `debug` would have provided.
- Reporter composition (`consola.addReporter(otelReporter)`, `consola.addReporter(jsonStdoutReporter)`) is a single concept all consumers already know.
- Bus saturation, async-await semantics, payload typing — all dodge-able by virtue of being "just a logger."

**Revisit if:** a non-log diagnostic event (structured counter, metric span) needs a typed surface — by then there's a real consumer to design against.

### D-019 — Adapter-private user hooks via `AdapterHooks<T>` on each adapter

**Decision:** Today's `puppeteer:before-goto` and `authenticate` hooks (and any future adapter-private user-customizable hooks) live on the adapter's own `Hookable<TAdapterHooks>` instance, exposed via `adapter.hooks` (separate from `core.hooks`). The user config supplies them under a namespaced key (e.g. `hooks: { 'auditors/local': { 'before-goto': fn } }`). The preset's auditor resolver wires those into the adapter's `adapterHooks` option at construction.

**Reasoning:**
- These are user-customizable but adapter-specific: only `auditors/local` runs Puppeteer; only auditors that authenticate need `authenticate`. Putting them on the core HookMap would force every adapter to know about Puppeteer.
- The `debug?` callback can't replace them — `before-goto` mutates page state, it's not diagnostic.
- Adapter hooks have their own type per adapter; the contract is local. Adding/changing them is the adapter's SemVer concern, not core's.

**Revisit if:** an adapter-private hook becomes universally interesting (rare).

### D-019b — `mergeAuditors` / `raceAuditors` deferred from v1

**Decision:** Ship only `pick`-based composition (`routeAuditors({ pick })` + the named-helper pick functions) in v1. `mergeAuditors([a, b])` and `raceAuditors([a, b])` are dropped from v1 scope.

**Reasoning:**
- Merge semantics are non-trivial (which `lighthouseVersion` wins; how do categories union; CrUX field data + local lab — `opportunities` only exist on lab; `capturedAt` reconciliation).
- No real consumer today; ship when a third user complains and the use case is concrete.
- Race is similarly under-specified (cancel the loser? consume both quotas?).

**Revisit if:** a CrUX + local-lab fused report becomes a top-N CLI request.

### D-019c — Single error class with `.code` discriminant

**Decision:** All thrown errors from core/api/presets are `UnlighthouseError extends Error` with `.code: string` discriminant. Stable codes are constants exported from `@unlighthouse/contracts/errors` (`NOT_SUPPORTED`, `QUOTA_EXCEEDED`, `ACTIVE_SCAN_CONFLICT`, `CONFIG_INVALID`, `SCAN_NOT_FOUND`, etc.). No class hierarchy.

**Reasoning:**
- AI agents key on string codes (already exposed as `exitCodes` in commands). A class hierarchy gives them nothing extra.
- Error matching becomes `err.code === 'NOT_SUPPORTED'` — same shape across HTTP/MCP/CLI transports.
- Single class means one `instanceof` check, no JS class hierarchy traps across module boundaries.

**Revisit if:** a code namespace grows past ~50 codes and discoverability suffers (then add `.namespace`).

### D-020 — Config rules are host-specific, not core-specific

**Decision:** Imperative config-resolution rules (e.g. "if `site.includes('localhost')` then disable throttle"; "autodetect chrome path"; "scale puppeteer cluster to CPU count") live in the host preset's config module — for the CLI, `@unlighthouse/unlighthouse/config/resolve.ts`. `core` receives an **already-resolved** `UnlighthouseConfig` and runs Zod validation only.

**Reasoning:**
- Today's `packages/unlighthouse/src/resolveConfig.ts` is ~190 LOC where ~50 LOC is `defu` calls and ~140 LOC is Node-specific rules (chrome paths, puppeteer options, localhost detection). Those rules don't run in a Worker.
- D-011 ("defu owns merging; Zod is validation-only") only covers the merge layer. Rules are a third layer that must live somewhere.
- Putting rules in the preset means the CF Worker preset can have its own rules ("if R2 bucket binding absent, disable artifact storage") without polluting core.

**Revisit if:** a rule needs to consult adapter state (none known today).

### D-021 — Node 22 baseline kills `better-sqlite3` and `zod-to-json-schema`

**Decision (rolled in from harlan, 2026-05-13):**
- **`node:sqlite` is the v1.0 default sqlite driver** for `storage/drizzle`. `better-sqlite3` is dropped from `dependencies` entirely. Users who want it pass their own `Database` instance into `drizzleStorage({ driver })`.
- **`z.toJSONSchema()`** (Zod 4 built-in) handles all JSON Schema generation for the `manifest` command and parity tests. `zod-to-json-schema` is **not** a dep. No `contracts/manifest` subpath needed.

**Reasoning:**
- The 0.x #1 install failure is `better-sqlite3` native compile. Node 22's stable `node:sqlite` makes the native dep unnecessary for the default install path. The drizzle `node-sqlite` driver landed in mid-2024.
- Zod 4 absorbed JSON Schema generation into the core package. Saves ~30KB on Worker bundles and one peer dep.
- Both changes consolidate on stdlib + first-party features the runtime baseline already commits to.

**Revisit if:** drizzle's `node-sqlite` driver lags `better-sqlite3` on a feature the CLI needs (WAL tuning, custom collations) — then re-add `better-sqlite3` as an opt-in driver, not a default.

### D-022 — Remote-CDP auditors are one adapter (`cdp-connect`); perf scores are flagged unreliable

**Decision:**
- Collapse `auditors/browserless`, `auditors/cloudflare-rendering`, and any future remote-Chrome backend into a **single adapter `auditors/cdp-connect`** parameterised by `{ browserWSEndpoint, headers? }`. CF Browser Rendering (binding + REST), browserless.io, and a self-hosted puppeteer-cluster all instantiate the same adapter.
- Declare `cdp-connect.capabilities.reliablePerfScores = false`. Router pick functions and the assertion engine consult this; presets can downgrade-with-warning on perf categories when only `cdp-connect` auditors are available.
- Document the cost/limits matrix per backend in the adapter's README; the adapter itself doesn't enforce backend-specific quotas (that's `rateLimitedPick` + a host bucket).

**Reasoning:**
- Lighthouse perf metrics depend on a calibrated CPU/network baseline. Remote CDP adds variable RTT and shared-infra CPU jitter; LCP/TBT/SI drift run-over-run. (Verified against CF Browser Rendering 2026 docs; the same caveat applies to any remote-CDP backend.)
- One adapter is the skill rule's correct shape (two real backends = one seam). The differences (auth header, WS URL resolution, concurrency cap) are configuration, not separate code paths.
- Capability declaration is the architectural answer to "perf is unreliable here" — it surfaces the constraint at the type level so callers can route around it, instead of hiding it in adapter prose.

**Non-perf use cases** (SEO, accessibility, best-practices, link discovery, CrUX field-data fan-out) work fine on remote CDP — router can fan perf to `local` and the rest to `cdp-connect` via a category-aware `pick`.

**Revisit if:** CF (or another remote-CDP host) ships dedicated baseline hardware with documented perf-score parity — flip `reliablePerfScores` based on the `browserWSEndpoint` configuration.

## CF Durable Object hibernation pattern (Phase 5 reference)

WebSocket hibernation = DO sleeps with WS open at zero cost; CF wakes it on incoming message or external RPC.

### `ScanEventsDO` (one per scanId)

- `state.acceptWebSocket(ws)` on upgrade — hibernation-capable.
- Subscribers held by CF runtime, not in-memory; survive DO restart.
- `broadcast(event)`: persist via `state.storage` (SQLite-backed DO for indexed replay), then fan-out via `state.getWebSockets()`.
- `webSocketMessage` handles late-joiner replay requests (`{ replay: { from: seq } }`).
- Terminal events (`scan:complete`/`cancelled`/`error`) set a 5-minute alarm for auto-cleanup.

### Producer integration

```ts
await env.SCAN_EVENTS_DO
  .get(env.SCAN_EVENTS_DO.idFromName(scanId))
  .fetch('http://do/broadcast', { method: 'POST', body: JSON.stringify(event) })
```

DO auto-spawns on first access; named by `idFromName(scanId)` so each scan hits its own DO. CF wakes DO automatically.

### Why SQLite-backed DO

- Cheaper than KV-backed DO.
- Indexed event replay via `state.storage.sql`.
- New CF feature 2025; preferred default for new DO classes.

### Cost profile

- DO sleeping with N WS open: $0/hour.
- DO active for broadcast: milliseconds billed.
- 10-min scan, 500 events ≈ 500 brief wakes, negligible cost.

### Same pattern for `RateLimiterDO`

One DO per quota bucket (e.g. `psi:user-123`). `consume()` is an RPC; `remaining()` reads from DO storage. Hibernation-capable counter.
